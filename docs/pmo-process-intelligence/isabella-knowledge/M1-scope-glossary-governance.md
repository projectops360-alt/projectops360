## EN: What is the PMO Process Intelligence Command Center?

ProjectOps360° is building a new executive dashboard called the PMO Process Intelligence Command Center. It complements — never replaces — the current PMO Command Center home dashboard, which remains the default view. When the server-side flag `PMO_PROCESS_INTELLIGENCE_DASHBOARD_ENABLED` is on, owners and admins see a "Process Intelligence Beta" switcher in the current dashboard header and can move between both views in one click, keeping their organization context. The new module lives at `/process-intelligence` and shows how work actually flows across the portfolio using process mining over the Project Event Graph: dominant paths, variants, rework loops, waiting, bottlenecks, plus Earned Value financials (PV, EV, AC, CPI, SPI, TCPI, ETC, EAC, VAC), risk, capacity, dependency, benefit and strategy overlays, evidence-backed Isabella recommendations and a non-persistent What-if simulation. Design is data-first and professional: light background, minimal motion, no game-like visuals. Every alert or recommendation must show source, formula, timestamp, confidence and limitations — nothing is displayed without evidence. Simulations never change real data; persistence requires an explicit, permissioned change-control action.
Source: docs/product-brain/capabilities/CAP-047-pmo-process-intelligence-command-center.md.
Verify: with the flag on as owner/admin, open the home dashboard and use the Process Intelligence Beta switcher.

## EN: PMO process-mining glossary for this module

Case: one traceable execution instance (in v1, a project or milestone journey reconstructed from project_event_log events sharing a case id). Activity: a business step derived from event types (for example task started, risk registered, gate approved). Variant: a distinct ordered sequence of activities followed by one or more cases; variants are ranked by frequency, duration, rework and outcome. Dominant path: the most frequent variant. Rework loop: a repetition or return in the sequence (reopened task, repeated approval) — it is detected from events, never decorated by hand. Waiting time: elapsed time between an activity finishing and the next one starting. Bottleneck: an activity or handoff that concentrates waiting, queue or downstream impact; it is calculated, not assumed. Conformance: comparison of real execution against an expected reference when one exists. Drill-down: navigating organization → project → milestone while filters are preserved. Evidence package: the source events, projections, formulas, timestamps, assumptions and a data-quality score that justify any displayed number. Temporal order never implies causality: causal links exist only when recorded explicitly.
Source: CAP-047 §1/§4; src/lib/process-mining/variants/engine.ts; src/lib/graph/between-analysis.ts.

## ES: Glosario de minería de procesos del PMO para este módulo

Caso: una instancia de ejecución trazable (en v1, el recorrido de un proyecto o hito reconstruido desde eventos de project_event_log que comparten un case id). Actividad: un paso de negocio derivado de tipos de evento (por ejemplo tarea iniciada, riesgo registrado, gate aprobado). Variante: una secuencia ordenada y distinta de actividades seguida por uno o más casos; las variantes se ordenan por frecuencia, duración, retrabajo y resultado. Ruta dominante: la variante más frecuente. Loop de retrabajo: una repetición o retorno en la secuencia (tarea reabierta, aprobación repetida) — se detecta desde eventos, nunca se decora a mano. Tiempo de espera: tiempo entre el fin de una actividad y el inicio de la siguiente. Cuello de botella: una actividad o traspaso que concentra espera, cola o impacto aguas abajo; se calcula, no se asume. Conformance: comparación de la ejecución real contra una referencia esperada cuando existe. Drill-down: navegar organización → proyecto → hito conservando los filtros. Paquete de evidencia: los eventos fuente, proyecciones, fórmulas, timestamps, supuestos y un puntaje de calidad de datos que justifican cualquier número mostrado. El orden temporal nunca implica causalidad: los vínculos causales solo existen cuando fueron registrados explícitamente.
Fuente: CAP-047 §1/§4; src/lib/process-mining/variants/engine.ts; src/lib/graph/between-analysis.ts.

## EN: Governance rules Isabella must respect in this module

The current dashboard is protected: it stays default and unchanged except for the approved switcher. The module only exists behind `PMO_PROCESS_INTELLIGENCE_DASHBOARD_ENABLED` (default OFF) and only owner/admin see it. Isabella may never show a recommendation without linked evidence, confidence and limitations, and may never fabricate either. What-if simulations are clearly separated from real data and never persist automatically. Isabella's knowledge for this module evolves only through versioned, tenant-safe packages with rollback — feedback never changes production behavior without a reviewable version, and model weights are never fine-tuned automatically. The Living Graph is not modified by this module. Financial numbers always show baseline version and status date, and actuals, commitments and accruals are never double counted.
Source: CAP-047 §2/§6/§7/§8.
