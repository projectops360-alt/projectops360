## EN: How the Process Intelligence module reads data

The module never queries database tables from its screens. It consumes versioned, typed contracts (`PMO_PI_CONTRACT_VERSION`) built by three pure engines. The flow projection (`buildFlowModel`) turns Project Event Graph cases into a process map: nodes are registry event types (activities) with frequency, average incoming waiting, rework occurrences and a calculated bottleneck score (waiting × frequency, normalized); edges connect consecutive activities and are marked as rework only when the target activity had already happened in the same case. Only BUSINESS_EVENT, non-compensating events are used, and the projection always reports honest quality numbers (events seen, used, excluded, cases without events, a 0–1 data-quality score). Variant discovery is delegated to the existing CAP-046 engine — the dominant path is the most frequent variant's activity sequence. Tenant isolation is defense-in-depth: RLS on the tables plus a mandatory `scopeToOrganization` barrier that drops any cross-tenant row before it reaches a contract. Temporal order never implies causality, and nothing is displayed without an evidence package (source events, formulas, projections, timestamps, assumptions, limitations, data quality).

## ES: Cómo lee datos el módulo de Process Intelligence

El módulo nunca consulta tablas de base de datos desde sus pantallas. Consume contratos tipados y versionados (`PMO_PI_CONTRACT_VERSION`) construidos por tres motores puros. La proyección de flujo (`buildFlowModel`) convierte casos del Project Event Graph en un mapa de proceso: los nodos son tipos de evento del registro (actividades) con frecuencia, espera media de entrada, repeticiones de retrabajo y un puntaje de cuello de botella calculado (espera × frecuencia, normalizado); las aristas conectan actividades consecutivas y se marcan como retrabajo solo cuando la actividad destino ya había ocurrido en el mismo caso. Solo se usan eventos BUSINESS_EVENT no compensatorios, y la proyección siempre reporta calidad honesta (eventos vistos, usados, excluidos, casos sin eventos, puntaje 0–1 de calidad de datos). El descubrimiento de variantes se delega al motor existente de CAP-046 — la ruta dominante es la secuencia de actividades de la variante más frecuente. El aislamiento de tenant es defensa en profundidad: RLS en las tablas más una barrera obligatoria `scopeToOrganization` que descarta cualquier fila de otro tenant antes de llegar a un contrato. El orden temporal nunca implica causalidad, y nada se muestra sin paquete de evidencia (eventos fuente, fórmulas, proyecciones, timestamps, supuestos, limitaciones, calidad de datos).

## EN: Financial formulas and boundaries the module obeys

All Earned Value math is delegated to the canonical financial library — one source of formulas: CV = EV − AC; SV = EV − PV; CPI = EV / AC; SPI = EV / PV; TCPI = (BAC − EV) / (BAC − AC); ETC = (BAC − EV) / CPI when no bottom-up estimate exists; EAC = AC + ETC (or AC + bottom-up ETC when supplied — ETC and EAC always use the same method); VAC = BAC − EAC. BAC prefers the current baseline over the original budget. Missing inputs produce an explicit "unavailable" metric with a reason — never an invented number. Every snapshot carries its baseline version and status date, and known limitations are declared (for example, SPI is not a calendar delay and EVM does not establish causality). The double-counting guard is structural: actual costs, commitments and accruals are separate fields end to end and the module never exposes an aggregate that sums them. Baseline history is immutable in the canonical model (financial_baseline_versions); the module only reads it.

## ES: Fórmulas financieras y límites que el módulo obedece

Toda la matemática de valor ganado se delega a la librería financiera canónica — una sola fuente de fórmulas: CV = EV − AC; SV = EV − PV; CPI = EV / AC; SPI = EV / PV; TCPI = (BAC − EV) / (BAC − AC); ETC = (BAC − EV) / CPI cuando no hay estimado bottom-up; EAC = AC + ETC (o AC + ETC bottom-up cuando se suministra — ETC y EAC usan siempre el mismo método); VAC = BAC − EAC. BAC prefiere la baseline vigente sobre el presupuesto original. Los insumos faltantes producen una métrica "unavailable" explícita con su razón — nunca un número inventado. Cada snapshot lleva su versión de baseline y fecha de corte, y declara sus limitaciones conocidas (por ejemplo, SPI no es un atraso de calendario y el EVM no establece causalidad). La protección contra doble conteo es estructural: costos reales, compromisos y devengos son campos separados de punta a punta y el módulo nunca expone un agregado que los sume. El historial de baselines es inmutable en el modelo canónico (financial_baseline_versions); el módulo solo lo lee.
