## EN: Realtime, reliability and data-source guarantees

Process Intelligence has consumed REAL data since its first adapter — there was never mock data to replace: events come from project_event_log, financials from financial_project_cockpit, risks/dependencies from their canonical tables and capacity from the canonical engine, all through RLS-scoped reads plus the org/project defense-in-depth barriers. Live updates use the signature pattern: the client polls a cheap RLS-scoped aggregate (event count + max sequence for the current scope) every 20 seconds and re-renders ONLY when the signature actually changes — no render storms, no websocket fan-out. On errors the poll backs off deterministically (20s doubling to a 120s cap) and the indicator says "live updates degraded" in text; polling pauses while the browser tab is hidden and every timer is cleaned up on unmount, so there are no subscription leaks. A "live · synced" indicator with the last sync time is always visible. Server-side observability is one deterministic JSON log line per projection build (events, cases, nodes, edges, truncation, duration — no personal data), following the Living Graph observability pattern. Performance is met by design: the canvas is static SVG with zero animation, layout is memoized, the min-frequency level-of-detail filter keeps 200-node graphs usable, and the module loads through route-level code splitting. There is no additional tracking vendor: the only durable telemetry is the audited recommendation feedback.

## ES: Realtime, fiabilidad y garantías de fuentes de datos

Process Intelligence consume datos REALES desde su primer adaptador — nunca hubo datos mock que reemplazar: los eventos vienen de project_event_log, las finanzas de financial_project_cockpit, los riesgos/dependencias de sus tablas canónicas y la capacidad del motor canónico, todo mediante lecturas con RLS más las barreras de defensa en profundidad por organización/proyecto. Las actualizaciones en vivo usan el patrón de firma: el cliente consulta un agregado barato con RLS (conteo de eventos + secuencia máxima del alcance actual) cada 20 segundos y re-renderiza SOLO cuando la firma realmente cambia — sin tormentas de render, sin fan-out de websockets. Ante errores, el sondeo retrocede deterministamente (20s duplicando hasta un tope de 120s) y el indicador dice "actualizaciones en vivo degradadas" en texto; el sondeo se pausa mientras la pestaña está oculta y todos los timers se limpian al desmontar, así que no hay fugas de suscripciones. Un indicador "en vivo · sincronizado" con la hora del último sync siempre es visible. La observabilidad del servidor es una línea JSON determinista por construcción de proyección (eventos, casos, nodos, aristas, truncamiento, duración — sin datos personales), siguiendo el patrón de observabilidad del Living Graph. El rendimiento se cumple por diseño: el canvas es SVG estático sin animación, el layout está memoizado, el filtro de nivel de detalle por frecuencia mínima mantiene usables grafos de 200 nodos, y el módulo carga con code splitting por ruta. No hay ningún proveedor de tracking adicional: la única telemetría durable es el feedback auditado de recomendaciones.
